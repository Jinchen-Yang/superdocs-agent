# superdocs · 工程质量评估报告

> 评估对象:`superdocs-agent`（北邮智能助手,commit `2dcf090`)
> 评估方式:5 个独立 AI 审计 agent 并行只读审计(安全 / 稳定性 / 前端 UX / 功能完整性 / 工程规范),逐文件核对,未改动任何代码。
> 评估日期:2026-06-27
> 技术栈:Mastra(Hono/Node) · PostgreSQL/pgvector · MiniSearch + jieba · AI SDK(多模型) · React 19 + Vite + Tailwind v4 + assistant-ui

---

## 一、总评

| 维度 | 评级 | 一句话结论 |
|---|---|---|
| 🔐 安全 | **C(中-偏弱)** | 地基对(SQL 全参数化、HMAC 会话、scrypt),但信任边界有系统性缺陷,门禁/限流/权限三道防线均可被绕过 |
| 🛡️ 稳定性 | **C(能用但脆弱)** | 用户级错误处理讲究,但进程级韧性差,有两枚"反向 SSH 隧道拓扑下几乎必触发"的崩溃定时炸弹 |
| 🎨 前端 UX | **B−** | 视觉/移动端打磨出色,但可访问性几乎从零 + 错误/加载反馈大面积静默吞掉 |
| ✅ 功能完整性 | **约 82%** | 核心链路全部真实落地、数据规模属实;扣分在"多模态名不副实"、账号生命周期缺口、文档脱节 |
| 🧱 工程规范 | **B−** | 代码克制、注释到位,但零测试、零 lint、CI 近乎空转,缺自动化护栏 |

**一句话:** 这是一个**完成度和用心程度都明显高于同体量学生项目**的作品——核心功能真实可用、安全意识在线、视觉细腻;但它目前是"一个人能跑起来的高质量原型",距离"可多人协作、可安全上线、可长期维护的工程"还差**一层自动化护栏(测试/lint/CI)** 和**几个上线前必修的安全/稳定性硬伤**。

## 二、🔴 上线前必修(阻塞级,按优先级排序)

这些是跨维度交叉确认、影响最大的问题,建议**逐条修复后再扩大使用范围**。

### P0-1. `X-Forwarded-For` 可伪造 → 校园门禁 + 限流双双失效【安全·根因】
- **定位**:`src/mastra/auth/ratelimit.ts:21-25`(`clientIp()`),被 `campus.ts:78`、`auth.ts:25/57/77` 引用
- **问题**:`clientIp()` 直接取客户端发来的 `x-forwarded-for` 第一段,无任何"可信反代跳数"校验。攻击者发 `X-Forwarded-For: 211.68.0.1`(任一校园段)即被判为校园网内 → **门禁形同虚设**;伪造不同 XFF 使限流 key 每次不同 → **暴力破解防线失效**(还会消耗真实北邮 SSO 登录尝试)。
- **修复**:不信任原始 XFF。Node 侧取 TCP 源 IP,仅当来源是已知反代白名单时才解析 XFF(从右往左数第 N 跳);nginx 用 `X-Real-IP` 且应用层只认该头。

### P0-2. pg.Pool 缺 `error` 事件监听 → 单条空闲连接被掐断即崩全站【稳定性】
- **定位**:`src/mastra/db/pool.ts:11-16`
- **问题**:`new Pool(...)` 后从未 `_pool.on('error', ...)`。pg 官方明确:空闲连接在后台被断开时发 `error` 事件,无监听器 → Node 按未捕获异常处理 → **整个进程 crash**。在"反向 SSH 隧道连 PG"的拓扑下,隧道抖动 / PG 重启 / NAT idle 超时**几乎必然触发**。
- **修复**:`_pool.on('error', e => console.error('[pg] idle client error', e))`;加 `keepAlive: true` 并配 PG 侧 keepalives。

### P0-3. 文档索引在 import 期同步读文件且无兜底 → 数据缺失即服务起不来【稳定性】
- **定位**:`src/mastra/tools/documents/search-index.ts:7-8,37`
- **问题**:`JSON.parse(readFileSync(META_PATH))` 在模块顶层执行且**无 try/catch**(隔壁 `knowledge-index.ts:21-27` 却做了降级)。rsync 同步半截 / 路径错配 → **整个 Mastra 进程 import 失败**。项目记忆里专门记了"数据产物走 rsync 的坑"。
- **修复**:与 knowledge-index 对齐,try/catch 降级为空索引 + 告警,让服务带空索引启动而非彻底拒服。

### P0-4. 会话不可撤销 / 登出无效 / 改密后旧 token 仍有效【安全】
- **定位**:`src/mastra/auth/session.ts:25-51`、`routes/auth.ts:97-100`
- **问题**:无状态 token = `userId.exp.hmac`,**有效期 30 天**,服务端无吊销表。token 一旦被窃,30 天内任意重放,登出 / 改密都挡不住;token 内无版本号,改密后旧 token 依然有效。
- **修复**:引入 `app_user.session_epoch`,token 内嵌 epoch,改密/登出时 +1 使旧 token 失效;或改短期 token + 刷新机制。

### P0-5. 管理员可由"用户名"判定 → 潜在提权【安全】
- **定位**:`src/mastra/auth/admin.ts:10-15` + `provider.ts:75-106`
- **问题**:`isAdmin` 同时接受 `external_id`(学号)**或** `username`,而 username 是用户注册时自选的任意字符串。若 `ADMIN_IDS` 配成像用户名的值,**任何有北邮 SSO 账号的人都能抢注该用户名拿到管理员权限**。
- **修复**:管理员判定只认 `external_id`(学号),或单独建 `is_admin` 字段,杜绝用用户可控字段判权限。

### P0-6. 限流可被主动重置 + 单进程内存态【安全·稳定性】
- **定位**:`src/mastra/auth/ratelimit.ts:3,16`
- **问题**:`if (hits.size > 10_000) hits.clear()` —— 攻击者用伪造 XFF 灌入上万个 key 即可**主动触发全量清空**,把撞库计数器清零;且 Map 为单进程内存,多实例/重启即失效。
- **修复**:换 Redis 共享存储 + 漏桶/固定窗口;淘汰用 LRU/按 key 过期而非整体 `clear()`;登录失败按账号维度也加锁。

## 三、按维度详细发现

### 🔐 3.1 安全(评级 C)

**🟠 高**
- **跨用户 thread 写入/记忆污染(IDOR)** — `routes/chat.ts:71-76`:归属校验依赖 `getThreadById` 的过滤语义,若该方法对他人 thread 返回 `null`,则可能走到 `saveThread({id: 他人threadId, resourceId: 自己})` 接管他人会话。需结合 Mastra 源码确认 `getThreadById`/`saveThread` 语义后定级。`conversations.ts` 同模式。

**🟡 中**
- **未设 `NODE_ENV=production` 时退回硬编码弱密钥 + 明文 cookie** — `session.ts:8-19`:fail-fast 只在 `NODE_ENV==='production'` 触发,线上漏配则用 `'dev-insecure-session-secret-change-me'` 签名(任何人可伪造 token)且关闭 `Secure`。建议以"`APP_SESSION_SECRET` 是否存在且够长"为判据。
- **用户名枚举(时序 + 错误路径)** — `provider.ts:62-72`、`password.ts`:用户不存在时立刻 return(不跑 scrypt),响应时间差暴露用户名是否存在;409"已被占用"也确认存在。
- **scrypt 用默认 cost 参数** — `password.ts:7,16`:未显式设 `N/r/p` 且未把参数编入存储串,未来 Node 改默认值会导致旧 hash 不可比对。
- **错误信息直接回显下游异常** — `auth.ts:16`、`profile.ts:66`、`chat.ts:81/106`:透传底层 message 可能泄露内部拓扑/库版本。
- **CORS 未显式配置** — `index.ts:33`:依赖 Mastra/Hono 默认,需确认默认 origin/credentials 行为。

**🔵 低**
- `whoami` 无需登录即泄露客户端 IP + 门禁开关 + 网段数(`auth.ts:108-116`,配合 P0-1 便于侦察)
- 无账号级锁定、密码仅校验长度 ≥8(`auth.ts:35`)
- 自我作用域的持久化 prompt 注入(`memory/memory.ts`,影响限用户自身,风险低)
- ✅ **正面确认**:SQL 全部参数化(`db/` + 各路由),未见拼接;HMAC + `timingSafeEqual`;cookie `HttpOnly/SameSite=Lax`;下载链接为固定白名单 host(无 SSRF 扩散面)

> ⚠️ 依赖漏洞扫描(`npm audit`)因私有 registry 未能运行,**需在可联网环境补做**。

### 🛡️ 3.2 稳定性 / 可靠性(评级 C)

> (🔴 严重项见第二节 P0-2 / P0-3)

**🟠 高**
- **大模型调用全程无超时/重试/降级** — `registry.ts:32-47`、`chat.ts:85,97`:provider 用默认 fetch 无 AbortController,某上游 hang(非报错)时请求无限期占住连接和内存,无法自动切其他模型。
- **`docsAgent.stream()` 未包 try/catch** — `chat.ts:85`:流初始化阶段抛错会冒泡成裸 500,与全局结构化中文错误风格不一致。
- **鉴权依赖 DB 且不容错 + 每次请求都打库** — `guard.ts:11-19`、`user.ts:16-30`:`currentUser` 每次 `ensureSchema()`+查库,DB 抖动直接放大成全站登录态失效。
- **缺健康检查与可观测性** — `index.ts`:无 `/health`、无请求日志、无指标、无错误上报,出问题只能靠用户报障。
- **SSO/OCR 外部调用无超时** — `provider.ts:13-40`:`buptLogin`/`ocrCaptcha` 两处 fetch 无超时,OCR 还未判 `r.ok` 直接 `.json()`;3 次重试叠加放大等待。

**🟡 中**
- 多个互不协调的连接池(自建 pool + 2 个 PostgresStore + PgVector)可能打爆 PG `max_connections`(`pool.ts:13`、`memory.ts:8-9`、`index.ts:32`)
- 无 `statement_timeout`,慢查询长期占用连接(`pool.ts:11-16`)
- 管理统计对 append-only 的 `app_usage` 全表聚合且无 `model` 索引,随时间无界变慢(`admin.ts:15-34`、`schema.ts:20-31`)
- CI 只 typecheck 且用 `npm install` 不可复现(`ci.yml:20-22`)
- 内存限流 + 内存索引无法水平扩展(`ratelimit.ts`、`search-index.ts`/`knowledge-index.ts`)
- 个人记忆"读改写"与 Agent 写入存在竞态 last-write-wins(`profile.ts:56-69`、`memory.ts:14-27`)

**🔵 低**
- `DATABASE_URL` 处理不一致(`pool.ts` 有清晰报错 vs `memory.ts:5`/`index.ts:32` 用 `!` 断言)
- token 用量记录失败完全静默(`chat.ts:35-47,87`,管理/计费数据可能悄悄缺失)
- `ensureThreadTitled` 的 TOCTOU(`chat.ts:19-32`,后果轻微)

### 🎨 3.3 前端 UX / 可访问性(评级 B−)

**🔴 严重(可访问性硬伤,工作量小、收益大)**
- **全局禁用缩放** — `web/index.html:6`:viewport 含 `maximum-scale=1`,低视力用户无法放大正文/公式/真题图(WCAG 1.4.4 硬性失败)。输入框 16px 字号本身即可避免 iOS 聚焦放大,不需禁缩放。
- **所有 Modal/抽屉缺焦点陷阱、Esc 关闭、对话框语义** — `AccountModal.tsx:44-101`、`AdminModal.tsx:30-87`、`Sidebar.tsx`(移动抽屉)、`AuthGate.tsx:79`:无 `role="dialog"`/`aria-modal`/焦点管理,键盘与读屏用户被困在背景。建议封装统一 `Dialog` 组件。
- **图标按钮普遍无 `aria-label`** — `Topbar.tsx:24/85`、`Thread.tsx:104-115`、关闭 X、`Sidebar.tsx:68`:读屏只读"按钮"无内容。

**🟠 高(错误/加载反馈系统性缺失)**
- 打开历史会话失败时 `catch {}` 静默吞错 → 留空白页/闪回欢迎页(`useChatController.ts:139-152`)
- 初始鉴权与各数据加载全程无反馈,loading 态纯空白,`me`/`models`/`conversations` 失败全静默(`App.tsx:111-118`、`32-33`、`57-77`)
- 保存记忆无成功/失败反馈且不校验 `r.ok`,给"已保存"错觉(`AccountModal.tsx:28-32` + `api.ts:48-53`)
- 流式回答无重试/重新生成,错误文本被拼进消息正文当 Markdown 渲染,助手气泡无复制按钮(`useChatController.ts:84-106`、`Thread.tsx:77-86`)
- Enter 键绕过 disabled 造成重复提交登录/注册(`AuthGate.tsx:59-64`)

**🟡 中**
- `outline-none` 抹掉焦点环且无替代(多处),键盘看不到焦点(WCAG 2.4.7)
- 流式输出无 `aria-live`,读屏不播报(`Thread.tsx`)
- 移动断点 760 与 Tailwind `md`(768) 不一致 → 761–767px 布局错位(`App.tsx:22,52`)
- Markdown 外链同标签页打开,点来源即跳走丢会话(`markdown-text.tsx:8-15`)
- 动画全量无视 `prefers-reduced-motion`(`index.css:77-80`、`Background.tsx`)
- 残留无用字段 `displayName`(`AuthGate.tsx:11`)

**🔵 低**:弱对比度次要文字、长列表无虚拟化(预防项)、`window.confirm` 体验割裂、主题首次加载忽略系统偏好

> ✅ 做得好的:玻璃拟态视觉、安全区/iOS 键盘处理(`--kb` 方案)、桌面/移动抽屉切换——同类学生项目中少见的细腻。
